# BOONWAVE — заметки по внедрению модели рабочего процесса

## Совместимость с текущим runtime

Модель встроена в существующую архитектуру без отдельного параллельного слоя.

- `process` остаётся одним из типов общей модели `Node`.
- Существующий массив `tasks` сохранён и расширен, а не заменён.
- Старые задачи с полями `text` и `completed` продолжают нормализоваться.
- Новые поля процесса добавляются через `TYPE_DEFAULTS` и `normalizeProcessData`.
- Существующее IndexedDB-хранилище пока не требует новой object store: дочерние сущности сохраняются внутри `node.data`.
- Отдельные stores для этапов, расходов и назначений материалов рассматриваются только после профилирования и утверждения миграционной стратегии.

## Что уже интегрировано

- `NODE_SCHEMA_VERSION` повышена до 6.
- `TASK_SCHEMA_VERSION` повышена до 2.
- Задача получила `processId`, `stageId`, `order`, lifecycle и назначения материалов.
- Процесс получил этапы, расходы, участников, назначения материалов, выбранный этап и режимы просмотра.
- Нормализация получает `nodeId`, поэтому дочерние сущности знают родительский процесс.
- Выбранный несуществующий или архивный этап автоматически заменяется первым активным.

## Что не входит в этот шаг

- UI полноэкранной карточки;
- drag-and-drop;
- service-команды сохранения;
- command history;
- миграция IndexedDB на отдельные stores;
- создание дочернего процесса и связи на холсте;
- archive/trash UI.

## Следующий технический шаг

1. Добавить чистые команды изменения `process.data`.
2. Обернуть их в service-слой с атомарным сохранением карточки.
3. Реализовать выбор этапа и фильтрацию задач/расходов в presenter.
4. После этого подключать UI и жесты перемещения списка.
