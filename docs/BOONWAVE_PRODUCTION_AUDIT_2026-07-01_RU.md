# BOONWAVE — production-аудит

Дата: 2026-07-01  
База аудита: ветка `main`, контрольная версия Core Quality №634  
Цель: сохранить текущий дизайн и логику, довести проект до полноценного iOS-приложения без повторной архитектурной пересборки.

## Итоговый статус

BOONWAVE сейчас является устойчивым production-oriented web core, но ещё не является готовым App Store-приложением.

Архитектура пригодна для дальнейшего развития без возврата к старым монолитным PWA-сборкам. Однако native-слой, эксплуатационная надёжность и функциональная полнота ещё не завершены.

## Сильные стороны

1. Единая доменная модель карточек и связей.
2. Модульные контроллеры рабочего пространства, карточек, связей, панели и хранения.
3. Централизованная gesture state machine.
4. Проверенное на iPhone управление: pan, drag, pinch, single tap, блокировка карточек.
5. Транзакционное IndexedDB-хранилище карточек, связей и медиа.
6. Версионируемые миграции базы данных.
7. Разделение метаданных медиа и Blob-хранилища.
8. Автоматические syntax/unit/integration проверки.
9. Каноническая ветка `main` и сохранённый архив старой v8.
10. Один service worker и автоматическая очистка старых BOONWAVE-кэшей.

## Исправленные в ходе аудита ошибки

- устранено одновременное существование активных `sw.js` и `sw-v2.js`;
- приложение переведено на один канонический `sw.js`;
- удалён versioned service worker;
- добавлена проверка отсутствующих файлов и устаревших артефактов;
- CI запускает полный `npm run check`;
- CI распространяется на `main` и все ветки `codex/**`;
- увеличен срок хранения диагностического артефакта CI;
- устаревший README заменён актуальным статусом проекта.

## Найденные риски

### Критичные до native-упаковки

1. Native storage adapter отсутствует. На native-платформе сейчас предусмотрен fallback на IndexedDB.
2. Нет `capacitor.config.*`, каталога `ios/`, Xcode-проекта и схем подписи.
3. Нет SQLite-миграций и теста эквивалентности IndexedDB ↔ SQLite.
4. Нет пользовательского механизма резервной копии и восстановления всей рабочей области.
5. Нет законченного жизненного цикла Archive → Trash → Restore/Permanent Delete.
6. Undo/Redo присутствуют в интерфейсной модели, но ещё не реализованы как журнал команд.

### Высокие

1. Нет browser E2E-набора для реального WebKit/iPhone-подобного поведения.
2. Часть CSS полноэкранной карточки создаётся из JavaScript; её нужно вынести в статический stylesheet до дизайн-финализации.
3. Нет общего реестра функций с критериями готовности.
4. Нет crash/error reporting и диагностического экспорта.
5. Нет политики квот и очистки больших медиа.
6. Нет отдельного теста обновления данных между версиями приложения на реальном устройстве.
7. Нет `package-lock.json`; установка зависимостей пока не полностью воспроизводима.

### Средние

1. README до аудита описывал старый PWA-патч, а не текущую архитектуру.
2. Имена некоторых файлов содержат `v2/v3`; это не runtime-конфликт, но до релиза их следует нормализовать.
3. Accessibility частично предусмотрена, но не пройдена полноценная проверка VoiceOver и Dynamic Type.
4. Нет локализационной архитектуры.
5. Нет privacy manifest, privacy policy, support URL и App Store metadata.

## Соответствие конечной цели

Проект соответствует плану по следующим основаниям:

- web и будущий iOS используют одну доменную и UI-логику;
- хранилище уже отделено интерфейсом адаптера;
- жесты централизованы;
- карточки не зависят от старого монолитного runtime;
- CI и миграции существуют;
- дизайн может развиваться поверх стабильной модели без полной пересборки.

Проект перестанет соответствовать плану, если:

- native SQLite будет добавлен напрямую в контроллеры вместо адаптера;
- появятся отдельные несовместимые web/iOS UI-ветки;
- дизайн карточек будет финализирован до утверждения их информационной модели;
- архив, корзина и undo будут реализованы как несвязанные локальные обработчики;
- новые gesture-сценарии будут добавляться вне GestureMachine/CardController.

## Решение по App Store

Упаковку в Capacitor можно начинать после завершения этапов функционального ядра, backup/restore и browser E2E. Публикацию в App Store можно начинать только после native SQLite, privacy, Xcode CI/manual QA и проверки миграций.

Текущий вердикт: архитектурная база пригодна; App Store readiness пока не достигнута.
