# Проверка этапа внедрения рабочего процесса

## Плюсы

- модель расширяет существующий `process` Node без параллельной архитектуры;
- старые задачи продолжают нормализоваться;
- операции сохраняют карточку до изменения store;
- сбой записи не меняет состояние интерфейса;
- перенос задачи сохраняет вложения и остальные данные;
- архивирование не удаляет данные физически;
- логика покрыта domain, integration и service-тестами.

## Минусы

- UI и presenter ещё не подключены;
- undo/redo для команд пока отсутствует;
- расходы, участники и материалы пока не получили полный набор service-команд;
- ветвление процесса ещё не реализовано транзакционно.

## Риски

- нельзя подключать drag-and-drop до проверки конфликта с вертикальной прокруткой и жестами холста;
- нельзя добавлять окончательное удаление до реализации корзины и подтверждения;
- нельзя создавать дочерний процесс несколькими независимыми записями без одной транзакции;
- перед merge обязателен полный `npm test` и проверка GitHub Actions.

## App Store направление

Текущий шаг соответствует production-направлению: логика отделена от UI, операции тестируемы, storage остаётся адаптерным, а изменения данных не зависят от DOM. Это позволяет позднее использовать тот же domain/service слой с Capacitor и SQLite adapter.
